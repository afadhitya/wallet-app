# Contributing to Wallet App

Thank you for your interest in contributing!

## Development Setup

### Prerequisites

- Go 1.25+
- [sqlc](https://sqlc.dev) for SQL code generation: `brew install sqlc` or `go install github.com/sqlc-dev/sqlc/cmd/sqlc@latest`
- [golangci-lint](https://golangci-lint.run) for linting: `brew install golangci-lint`

### Clone and build

```sh
git clone https://github.com/afadhitya/wallet-app.git
cd wallet-app
make build
```

### Running tests

```sh
make test             # Run all unit tests
make test-cover       # Run tests with coverage profile and HTML report
make coverage-check   # Run tests and enforce 100% coverage
```

The project enforces 100% test coverage in CI. A small set of infrastructure-level branches (OS/filesystem errors, data integrity guards) are excluded via `coverignore.txt`.

## Project Structure

```
wallet-app/
├── cmd/
│   └── wallet/main.go         # Entry point
├── internal/
│   ├── cli/                   # Cobra CLI commands (presentation layer)
│   │   ├── helpers.go         # Service wiring, JSON output, error classification
│   │   └── *.go               # One file per command group
│   ├── service/               # Business logic layer
│   │   ├── service.go         # Service struct, error types
│   │   └── *.go               # One file per domain
│   ├── db/                    # Database infrastructure
│   │   ├── db.go              # Connection and migration runner
│   │   └── migrations/        # Versioned SQL migration files
│   ├── query/                 # sqlc query definitions (raw SQL)
│   ├── gen/                   # sqlc-generated Go code (do not edit)
│   └── testdb/                # In-memory SQLite for tests
├── pkg/
│   └── config/                # TOML configuration loading
└── brainstorming/             # Design documents
```

The architecture follows a layered pattern: **CLI → Service → Generated Queries → SQLite**.

- **`cmd/`** — Entry point only; creates and executes the root Cobra command
- **`internal/cli/`** — Cobra commands that parse arguments and delegate to the service layer
- **`internal/service/`** — Business logic, validation, and orchestration. All database access goes through the `gen.Querier` interface
- **`internal/gen/`** — Auto-generated by sqlc from the SQL files in `internal/query/`. Never edit manually
- **`internal/db/`** — SQLite connection management and migration application
- **`pkg/config/`** — TOML config structures and file loading

## Coding Conventions

### Go

- Format with `make fmt` (runs `go fmt ./...`)
- Lint with `make lint` (runs `golangci-lint`)
- Structurally match the existing file-per-domain pattern in `internal/service/` and `internal/cli/`
- Use `int64` for all amounts — always store in **integer minor units** (e.g., Rp 50,000 → `50000`, $50.00 → `5000`). Never use floating point for financial values

### Database

- Schema changes go in a new numbered migration file: `internal/db/migrations/NNN_description.sql`
- Query definitions go in `internal/query/<domain>.sql`
- After modifying queries or schema, run `make sqlc-gen` to regenerate `internal/gen/`

### sqlc code generation

```sh
make sqlc-gen
```

This runs `sqlc generate` using the config in `sqlc.yaml`. The config reads schema from `internal/db/migrations/` and queries from `internal/query/`, then generates type-safe Go code into `internal/gen/`.

### CLI documentation generation

```sh
make docs
```

This generates Markdown CLI reference documentation from the Cobra command tree into `docs/cli/` (git-ignored). Run this after adding or modifying any CLI commands to update the reference docs. The generated docs are not committed — CI may generate them on tagged releases for publishing.

### Commits

Follow [Conventional Commits](https://www.conventionalcommits.org/):

- `feat:` for new features
- `fix:` for bug fixes
- `docs:` for documentation changes
- `refactor:` for code restructuring
- `test:` for test additions or changes
- `chore:` for maintenance tasks

## Pull Request Process

1. Create a feature branch from `main`
2. Make your changes with clear, conventional commits
3. If you added or modified CLI commands, run `make docs` to regenerate CLI reference documentation
4. Ensure `make coverage-check` passes (100% coverage required)
5. Ensure `make lint` passes with no warnings
6. Open a pull request with a descriptive title and summary of changes
7. All checks must pass before merge
8. PRs are merged via squash merge to keep history clean

## Testing Guidance

- Service layer tests use `internal/testdb/` which provides an in-memory SQLite with full migrations
- CLI integration tests override the service layer via `getServiceOverride` in `internal/cli/helpers.go`
- Each domain module has its own `_test.go` file alongside the implementation
- The coverage gate runs in CI via `make coverage-check`. Infrastructure-level exclusions are documented in `coverignore.txt`
